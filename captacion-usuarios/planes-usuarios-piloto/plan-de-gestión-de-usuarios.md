# Plan de Gestión de Usuarios Piloto y Validación Ágil - ROOMA

**Proyecto:** ROOMA — App para encontrar compañero de piso sin agencia + gestionar gastos compartidos  
**Fase:** Validación con usuarios piloto previa al lanzamiento  
**Periodo:** Abril - Mayo 2026  
**Relación con otros documentos:** Este plan se conecta con el informe de feedback de usuarios piloto, el plan de marketing, la campaña de lanzamiento y el análisis económico del proyecto.

---

## 1. Objetivo del plan

El objetivo de este documento es definir cómo se han captado, gestionado y utilizado los usuarios piloto de ROOMA durante la fase de validación del producto.

La validación con usuarios piloto permite:

- probar funcionalidades reales antes del lanzamiento;
- detectar fricciones de uso;
- validar si la propuesta de valor se entiende;
- recoger feedback cualitativo y cuantitativo;
- alimentar el backlog de mejoras;
- reducir el riesgo de lanzar un producto sin contraste real;
- obtener señales iniciales para marketing, precios y modelo económico.

---

## 2. Estrategia de adquisición de usuarios piloto

Para obtener feedback realista y accionable, se ha trabajado con un grupo cerrado de usuarios piloto. La captación se ha organizado según los dos perfiles principales de la plataforma.

### Perfil inquilino

Incluye estudiantes, jóvenes profesionales o personas que viven actualmente en pisos compartidos.

| Fuente de captación | Justificación |
|:---|:---|
| Compañeros de facultad | Público cercano al segmento universitario objetivo |
| Amigos y contactos personales | Acceso rápido a usuarios dispuestos a probar |
| Personas que viven en pisos compartidos | Feedback más realista sobre convivencia y gastos |
| Redes universitarias | Canal natural para validar demanda inicial |

### Perfil propietario

Incluye caseros particulares, pequeños arrendadores y contactos relacionados con vivienda.

| Fuente de captación | Justificación |
|:---|:---|
| Arrendadores actuales o conocidos | Acceso directo a necesidades reales de gestión |
| Contactos de propietarios | Validación del lado de oferta |
| Plataformas como Idealista | Referencia del mercado actual |
| Despachos o contactos inmobiliarios | Visión profesional sobre publicación y gestión |

---

## 3. Incentivos de participación

La participación de usuarios piloto se plantea como una colaboración limitada, ligera y útil para ambas partes.

| Perfil | Incentivo |
|:---|:---|
| Inquilino | Participación en sorteos o incentivos menores por completar pruebas |
| Propietario | Posible acceso a cuenta fundador o condiciones preferentes durante el lanzamiento |
| Usuario piloto activo | Reconocimiento como primer usuario validador del producto |

El incentivo no sustituye al feedback real. Su función es aumentar la participación y reducir el abandono durante las pruebas.

---

## 4. Gestión de datos y privacidad

La gestión de usuarios piloto se realiza bajo un criterio de minimización de datos. Solo se recopila la información necesaria para interpretar el feedback y clasificar correctamente las respuestas.

### Datos recopilados

| Dato | Uso |
|:---|:---|
| Nombre | Identificación básica del participante |
| Edad | Segmentación de perfil |
| Rol | Clasificar como inquilino, propietario o usuario mixto |
| Correo electrónico | Contacto para pruebas y encuestas |
| Experiencia previa | Saber si usa apps inmobiliarias, financieras o de convivencia |
| Feedback de uso | Mejorar producto, comunicación y flujos |

### Datos no recopilados durante el piloto

Durante la fase piloto no se recopilan:

- documentos de identidad reales;
- datos bancarios reales;
- contratos reales;
- pagos reales;
- datos sensibles no necesarios;
- información financiera privada.

Los flujos de pago, facturas o verificación se prueban mediante datos ficticios o simulados.

## 5. Acuerdo de compromiso con usuarios piloto

La participación se organiza mediante un compromiso simple entre el equipo de ROOMA y los usuarios piloto.

### Compromisos del equipo ROOMA

El equipo se compromete a:

- facilitar acceso al prototipo o versión beta correspondiente;
- explicar claramente qué debe probar el usuario;
- no solicitar datos sensibles innecesarios;
- usar el feedback únicamente para mejorar el producto;
- respetar la privacidad de los participantes;
- no saturar al usuario con comunicaciones excesivas.

### Compromisos del usuario piloto

El usuario piloto se compromete a:

- dedicar un tiempo reducido a probar el producto;
- responder encuestas o preguntas breves;
- comunicar errores, dudas o bloqueos;
- dar feedback honesto;
- probar los flujos asignados según su perfil.

El tiempo máximo estimado por prueba se mantiene entre **10 y 20 minutos** para no generar fricción ni abandono.

## 6. Metodología de validación

La validación se ha dividido en tres sprints, cada uno centrado en una parte concreta del producto.

| Sprint | Objetivo principal | Tipo de validación |
|:---:|:---|:---|
| Sprint 1 | Validar onboarding, perfil y matchmaking | Comprensión inicial y navegación |
| Sprint 2 | Validar chat, incidencias, facturas y flujo Self-Match | Usabilidad funcional y valor percibido |
| Sprint 3 | Validar flujo completo y estabilidad | Experiencia end-to-end y preparación para lanzamiento |

El Community Manager actúa como canal principal de comunicación para centralizar feedback, evitar mensajes duplicados y trasladar información útil al equipo de producto.

---

## 7. Sprint 1: Validación del core y matchmaking

### Objetivo

Validar si el usuario entiende la propuesta básica de ROOMA: crear perfil, descubrir otros perfiles o viviendas y generar una conexión mediante un sistema de match.

### Elementos probados

- Onboarding inicial.
- Creación de perfil.
- Preferencias básicas.
- Sistema de descubrimiento.
- Flujo de like/dislike.
- Comprensión del match.

### Métricas observadas

| Métrica | Finalidad |
|:---|:---|
| Tiempo hasta entender la interfaz | Medir claridad inicial |
| Finalización del flujo de perfil | Detectar fricción en onboarding |
| Comprensión del sistema de match | Validar propuesta central |
| Dudas repetidas | Identificar puntos confusos |

### Método de feedback

Se utiliza una prueba asíncrona con enlace al prototipo o versión disponible, seguida de un cuestionario breve con escala Likert y preguntas abiertas.

---

## 8. Sprint 2: Validación de funcionalidades diferenciales

### Objetivo

Validar si las funcionalidades posteriores al match aportan valor real al usuario y diferencian ROOMA frente a canales tradicionales.

### Funcionalidades evaluadas

- Chat post-match.
- Envío de fotos.
- Gestión de incidencias.
- Gestión de facturas.
- Simulación de pago.
- Flujo completo Self-Match.

### Métricas observadas

| Métrica | Finalidad |
|:---|:---|
| Facilidad percibida | Saber si el flujo se entiende |
| Confianza en facturas y pagos | Validar módulo económico |
| Utilidad de incidencias | Detectar valor diferencial |
| Comprensión del flujo completo | Medir continuidad de la experiencia |
| Disposición a pagar | Alimentar modelo de ingresos |

### Método de feedback

Se combina observación guiada, encuesta breve y preguntas abiertas. El objetivo no es solo saber si el usuario puede completar una acción, sino entender qué parte le genera confianza, duda o interés.

---

## 9. Sprint 3: Validación del flujo completo

### Objetivo

Validar la experiencia completa de ROOMA antes del lanzamiento, conectando los flujos principales en una secuencia coherente.

### Flujo validado

1. Registro o entrada en la app.
2. Creación o revisión de perfil.
3. Descubrimiento de perfiles o viviendas.
4. Match.
5. Chat.
6. Invitación o coordinación.
7. Gestión de factura o pago simulado.
8. Revisión de incidencias o convivencia.

### Métricas observadas

| Métrica | Finalidad |
|:---|:---|
| Finalización del flujo completo | Medir estabilidad y continuidad |
| Bloqueos técnicos | Detectar errores críticos |
| NPS inicial | Medir recomendación |
| Sugerencias abiertas | Alimentar backlog |
| Valor percibido | Validar si ROOMA se entiende como producto completo |

### Método de feedback

Se realiza una encuesta final de valoración global y se recopilan sugerencias para versión 1.0, lanzamiento y mejoras posteriores.

---

## 10. Tratamiento del feedback

Al finalizar cada sprint, el feedback se procesa para convertirlo en decisiones de producto, marketing y negocio.

### Tipos de feedback recogidos

| Tipo de feedback | Ejemplo | Uso |
|:---|:---|:---|
| Cuantitativo | Puntuaciones de 1 a 5 | Medir facilidad y aceptación |
| Cualitativo | Comentarios abiertos | Detectar dudas y mejoras |
| Técnico | Bugs o errores | Priorizar correcciones |
| Comercial | Disposición a pagar | Ajustar modelo de precios |
| Marketing | Mensajes que se entienden mejor | Mejorar comunicación |
| Producto | Funciones más valoradas | Priorizar backlog |

### Flujo de tratamiento

1. Recogida de respuestas.
2. Agrupación por tipo de usuario.
3. Separación entre bugs, dudas y sugerencias.
4. Priorización según impacto.
5. Comunicación al equipo de producto.
6. Incorporación al backlog.
7. Uso de aprendizajes en marketing y lanzamiento.

---

## 11. Priorización de mejoras

Las mejoras detectadas se clasifican según impacto y urgencia.

| Prioridad | Criterio |
|:---:|:---|
| Alta | Afecta al flujo principal o bloquea el uso |
| Media | Mejora claridad, confianza o retención |
| Baja | Mejora visual o detalle no crítico |

### Ejemplos de priorización

| Tipo | Ejemplo | Prioridad |
|:---|:---|:---:|
| Bug | El chat no permite enviar texto sin imagen | Alta |
| UX | Estados de incidencias poco claros | Media |
| Producto | Calendario integrado de pagos e incidencias | Media |
| Marketing | Mostrar mejor el flujo Self-Match | Media |
| Visual | Ajustes menores de textos o iconos | Baja |

---

## 12. Relación con marketing y lanzamiento

El feedback de usuarios piloto no solo sirve para mejorar la app. También alimenta directamente la comunicación del lanzamiento.

### Aprendizajes aplicados a marketing

| Aprendizaje | Aplicación |
|:---|:---|
| El usuario entiende mejor la app viendo el flujo completo | Crear demos cortas del recorrido Self-Match |
| Incidencias se percibe como función diferencial | Comunicar ROOMA como herramienta de convivencia |
| Facturas refuerzan utilidad posterior al match | Mostrar gestión de gastos como beneficio clave |
| Chat y match son fáciles de entender | Usarlos como entrada visual en piezas de lanzamiento |
| Hay sensibilidad al precio | Ajustar mensajes de Rooma Plus y Rooma Pro |

### Uso en campaña

Los resultados del piloto permiten preparar:

- testimonios breves;
- argumentos de confianza;
- copies más claros;
- demos de producto;
- respuestas a dudas frecuentes;
- contenidos centrados en problemas reales.

## 13. Relación con el análisis económico

La validación con usuarios piloto también afecta al modelo económico de ROOMA.

| Elemento económico | Relación con usuarios piloto |
|:---|:---|
| Rooma Plus | Se contrasta con la disposición a pagar del perfil inquilino |
| Rooma Pro | Se valida con propietarios y usuarios de mayor valor |
| Add-ons | Se evalúan como alternativa a suscripción completa |
| Módulo de facturas | Refuerza la vía transaccional |
| CAC combinado | Los pilotos y referidos pueden reducir dependencia de Ads |
| Retención | Las funciones post-match pueden aumentar uso recurrente |

La gestión de usuarios piloto reduce incertidumbre antes del lanzamiento y permite ajustar hipótesis de conversión, adopción y valor percibido.

---

## 14. Riesgos de la validación piloto

| Riesgo | Impacto | Mitigación |
|:---|:---|:---|
| Muestra pequeña | Puede no representar al mercado real | Complementar con lanzamiento público y métricas reales |
| Usuarios demasiado cercanos | Feedback menos crítico | Formular preguntas concretas y pedir objeciones |
| Baja participación | Falta de datos útiles | Reducir tiempo de prueba e incentivar participación |
| Feedback disperso | Dificultad para priorizar | Centralizar respuestas y agrupar por tipo |
| Pruebas con datos ficticios | No valida completamente pagos reales | Usar simulación inicial y validar pagos reales más adelante |
| Sesgo positivo | Usuarios dicen que usarían la app pero luego no vuelven | Medir retención y uso real en lanzamiento |

---

## 16. Conclusión

La gestión de usuarios piloto permite validar ROOMA antes de abrirlo a un público más amplio. El objetivo no es únicamente conseguir opiniones positivas, sino detectar fricciones, confirmar qué funcionalidades generan más valor y ajustar el producto antes del lanzamiento.

La validación realizada en los tres sprints aporta información útil para producto, marketing y análisis económico. Los usuarios piloto ayudan a comprobar si el flujo principal se entiende, si las funciones diferenciales tienen sentido y si existe disposición inicial a pagar por determinados planes.

A partir de esta validación, ROOMA puede llegar al lanzamiento con mayor claridad sobre sus puntos fuertes, sus riesgos y las mejoras que deben priorizarse para versión 1.0.
